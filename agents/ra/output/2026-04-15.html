The Story · CLAUDE.md Is a New Kind of Code

Yesterday, a GitHub repository called andrej-karpathy-skills picked up 9,263 stars in a single day. It is not a library. It is not a framework. It is one text file — a CLAUDE.md — distilling Andrej Karpathy's observations on how to get better results from AI coding agents. The same day, a Claude Code best practices repo added 2,583 stars, a context-injection plugin that automatically captures and reinjeets Claude Code session history added 2,997, and Claude Code Routines — scheduled automation triggers for Claude — landed as HN's #3 story with 611 points and 355 comments. NousResearch's Hermes Agent, positioning itself as "the agent that grows with you," pulled 8,301 stars overnight.

That's a pattern. The community is converging on a meta-layer: not what the AI does, but how it's set up to work.

A CLAUDE.md file is essentially a standing briefing document. It tells your agent who it is, what rules it follows, what mistakes to avoid, what context to carry forward. The Karpathy version earned nearly 10,000 stars in one day because people recognize that the configuration layer is where a lot of the performance difference lives. Two developers running the same model on the same codebase can get dramatically different results — not because of prompt tricks in the moment, but because of how the environment around the agent has been structured.

This represents a real shift in what "knowing how to use AI" means. A year ago the skill was prompt engineering — clever phrasing to get the model to cooperate in a single exchange. That's been superseded. The current skill is agent configuration: writing the standing instructions that make your agent reliably useful across thousands of interactions, not just one. It's closer to writing a job description than writing a prompt. The best CLAUDE.md files aren't just rules — they're a theory of how the agent should reason, what it should remember, and what it should refuse.

What the Hermes Agent signal adds to this: there's real appetite for agents that evolve their own configuration over time rather than requiring humans to update the file. Whether it delivers on that is an open question — 8,000 stars is a thesis, not a proof — but the direction is clear. The ceiling for a static CLAUDE.md is the ceiling for the person who last edited it. An agent that can update its own operating rules based on outcomes is a different class of system.

If you want to understand where agentic computing sits in its maturity curve, look at what the community is building around, not what the labs are shipping. Right now they're building around configuration, memory, and continuity. That is not beginner behavior. That is a community that has used the tools long enough to identify where the leverage actually is.

The Lab · What AI can newly do this week

Memory Worth: a primitive for forgetting well. A new arXiv paper proposes Memory Worth (MW) — a two-counter signal attached to each memory in an agent's store, tracking how often it was retrieved and how often that retrieval was actually useful based on downstream task outcomes. The core insight: write-time importance scores (how valuable something seemed when you saved it) are static, but memory quality degrades as the agent's task distribution shifts. A memory that was critical during one project phase actively misleads when the project has moved on.

MW is a governance layer that lets an agent age out degraded memories without human curation. Any long-running agent accumulates noise alongside signal; without a principled deprecation mechanism, the retrieval layer gets polluted over time. This is infrastructure-level work, not a plug-in, but it names a real problem in a precise way. Reach for this pattern when building agents that maintain episodic memory across many interactions — health agents, research agents, coding assistants that carry project context over weeks. Filed at kai/research/2026-04-15-memory-governance.md.

Memory as Metabolism: the personal wiki architecture. A companion paper surveys the emerging cluster of personal-wiki-style memory systems: Karpathy's notes, MemPalace, LLM Wiki v2. The common thread is treating memory not as a retrieval index but as a growing linked artifact that gets processed over time — knowledge is ingested, refined, and the unusable parts are dropped. What remains is the agent's evolving model of the world, not just a searchable archive. The timing is notable: this and the MW primitive both landed the same week as the Claude Code configuration wave. Researchers, practitioners, and architects are all converging on the same layer simultaneously. Filed at kai/research/2026-04-15-memory-architecture.md.

Worth Sitting With

A paper that got less attention than it deserves today: The Non-Optimality of Scientific Knowledge, which argues that the body of scientific knowledge at any given moment is a local optimum, not a global one. The frameworks we use to understand the world aren't the best possible frameworks — they're the ones we locked into given the trajectory we happened to take. Path dependence, not correctness, explains much of what we collectively believe.

This is uncomfortable in a specific way. Scientific consensus is the mechanism we rely on to correct bad ideas. But if the consensus is itself locked into a local minimum — a paradigm that's internally coherent but globally suboptimal — the correction mechanism fails quietly. Anomalies accumulate. The paradigm holds until someone builds a new one that explains them better.

What makes this timely is that we're crystallizing enormous amounts of scientific and engineering knowledge into model weights and training data right now. If the knowledge we're training on is itself path-dependent, what are we building? Systems that are very good at the local minimum we happen to be in? The question isn't whether to proceed — it's whether we're honest about what "state of the art" actually means. It means best we've found so far along the path we took, not best possible. That distinction matters more when the systems are making decisions than when the systems are just retrieving information.

What would it look like to train a model specifically on paradigm shifts rather than consensus knowledge?

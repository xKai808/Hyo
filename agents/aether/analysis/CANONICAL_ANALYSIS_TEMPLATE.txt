4.1 HARVEST MISSES — CONFIRMED ROOT CAUSE DISCREPANCY
------------------------------------------------------
The harvest miss diagnostic is now producing a new critical signal:

  → HARVEST MISS DIAG | yes_bids:ABSENT | no_bids:ABSENT

This appears on EVERY harvest miss across all 4 days. The parsed_ob shows
orderbook depth. The int_bdi shows real depth (e.g., 805, 1112, 863 on Apr 15).
But the exchange returns ABSENT bids when the order is placed.

Two distinct failure modes are observed:

MODE A — Low anchor_depth:
  int_bdi 11 | near_bdi 11 | anchor_depth(±0.02) 0
  (Apr 14, 01:35 event)
  Here the ±0.02 filter genuinely has no liquidity. The fix (anchor ±0.02
  instead of held_px ±0.05) would help. This is the original hypothesis.

MODE B — High BDI, deep book, still ABSENT:
  int_bdi 822 | anchor_depth(±0.02) 817 | yes_bids:ABSENT | no_bids:ABSENT
  (Apr 15, 00:42 event)
  817 contracts at anchor depth. Book is full. But exchange says ABSENT.
  The fix is NOT the depth filter — the depth is there. The problem is
  timing: the orderbook state at poll time differs from the state when
  the order hits the exchange. The book is moving between poll and order.

IMPLICATION: Two separate fixes are needed, not one.
  Mode A fix: anchor ±0.02 depth (already identified — Issue #1 current theory)
  Mode B fix: order placement timing — the harvest order is being placed
              at a price that existed during polling but is gone by fill time.
              This is a stale orderbook / race condition problem.

4.3 POS WARNING — NEW CRITICAL SIGNAL [FIRST APPEARANCE]
---------------------------------------------------------
New log event observed in Apr 16:
  → POS WARNING | API 0 local 2
  → POS WARNING | API 0 local 15

This fires during exit sequences. Meaning: AetherBot's local state says it
holds N contracts. The exchange API reports 0.

Observed in conjunction with:
  - STOP UNFILLED / book thin events
  - EXIT_ESCALATED_AFTER_BDI0 sequences
  - CONTRACT_VELOCITY exits that fail to fill

When the exchange says API 0 while local says 15, one of three things is true:
  (a) The original BUY order did not fill — AetherBot is trying to exit a
      phantom position.
  (b) The exchange already settled the position (contract expired) before
      AetherBot issued the exit — the state is stale.
  (c) There is an API polling lag where the position will appear on next poll.

Case (b) is most likely given that the POS WARNINGs appear at <120s remaining.
Case (a) would mean AetherBot enters a loss without ever having a real position —
this would be the most severe scenario and requires immediate investigation.

The v254 instrumentation MUST capture POS WARNING context:
  - What is the full API response when the position check returns 0?
  - Was the original BUY order confirmed by the exchange?
  - Does the POS WARNING resolve on the next poll or persist?

PATTERN 1 — BDI=0 HOLD WITH <120s REMAINING = EXPIRY LOSS
------------------------------------------------------------
Across 4 sessions: 4 confirmed cases where BDI=0 hold fired with <120s
remaining. In 3 of 4 cases, the position expired worthless or near-worthless.

The hold logic exists to prevent exiting into a momentary BDI collapse that
will recover. But with <120s left, recovery is impossible — the contract expires.
A time gate on BDI=0 hold behavior is warranted and well-supported by log evidence.

Mechanism question: Does the current BDI=0 hold code check seconds_left?
Log evidence suggests it does not. It holds regardless of contract time remaining.

PATTERN 2 — HARVEST MODE B (HIGH BDI + ABSENT BIDS) is not a depth problem
---------------------------------------------------------------------------
Apr 15 shows harvest misses with int_bdi 805, 863, 1112 — enormous depth.
anchor_depth(±0.02) shows 490, 817, 756 contracts at the target rung.
Yet yes_bids:ABSENT / no_bids:ABSENT on every attempt.

This is a market microstructure problem: the harvest order is being routed
to a price level where bids existed during the last poll but have been filled
or cancelled by the time the sell order hits the exchange. The ±0.02 filter
fix addresses Mode A (thin book) but NOT Mode B (stale book).

Mode B fix requires: place the harvest order at the CURRENT best bid at order
submission time, not the polled anchor price. The anchor is stale.

### 2. OB_FORMAT_DIAG REVEALED THE KALSHI SCHEMA  ← high-value intelligence
The whole point of v254's P1.2 diagnostic was to answer: "why does `yes_bids:ABSENT | no_bids:ABSENT` show up on 96/96 HARVEST MISS?" We now have the answer, printed 3 times in the log:

```
-> OB_FORMAT_DIAG | raw_keys: ['orderbook_fp'] | orderbook_keys: ['no_dollars', 'yes_dollars']
```

Decoded:
- Top-level API response key is `orderbook_fp` (fingerprint-wrapped), not a plain `orderbook`.
- Inner keys are `yes_dollars` / `no_dollars`, not `yes_bids` / `no_bids`.

**This means the OB parser has been looking for the wrong keys.** The harvest system's 11% success rate on 4/9 was not "Kalshi is down" and not "the parser crashed" — it was a **key name mismatch**. The parser was probably written against an older Kalshi API schema that exposed `yes_bids` / `no_bids` directly, and Kalshi has since changed to `yes_dollars` / `no_dollars` wrapped in `orderbook_fp`.

**v255 fix (spec seed, not code):**
- Update `get_orderbook()` to unwrap `orderbook_fp` before reading bids.
- Update harvest anchor derivation to read `yes_dollars` / `no_dollars` instead of `yes_bids` / `no_bids`.
- Keep `OB_FORMAT_DIAG` one-shot logging as a regression tripwire.
- Delete `OB_PARSER_DIAG` after v255 ships cleanly for 3 days (it was a diagnostic, not a feature).
- Expected impact: harvest success rate should jump from 11% → whatever the underlying signal quality supports. 4/8 showed functional harvesting, so the ceiling is probably 60-80%. Even at 50%, multi-day harvest P&L could contribute +$2-5/day.

**P0 status for v255.** This is a single structural bug with direct revenue impact. The fact that v254 diagnostics exposed it this cleanly means v254 earned its build cost already — the deploy paid for itself with data even before any trading-side fix fires.

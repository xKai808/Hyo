Chapter 01

What Is an AI Agent?

0:00 – 1:30

An AI agent is a system that uses a language model to make decisions and take actions in order to complete a goal — often one that requires multiple steps and involves the external world.

ChatbotAgent
Responds once per inputTakes sequences of actions
Reads only what you give itReaches out to external tools, APIs, files
No memory between turnsMaintains state across steps
Cannot change the worldCan write files, send emails, run code

The ReAct Loop — How an Agent Works

  1. Receive a goal. Given as a prompt from a human or another system with explicit success criteria.
  2. Reason about what's needed. The LLM generates a plan or next action based on the goal and all available context.
  3. Select a tool or action. The model chooses from available tools (web search, calculator, code runner, file reader, etc.).
  4. Execute the action. The tool runs and returns an observation.
  5. Incorporate the observation. The result is added to the agent's context.
  6. Repeat steps 2–5 until the goal is complete or a stopping condition is met.
  7. Return the result. The agent produces a final output and stops.

This loop is Reason → Act → Observe → Reason again. The loop name (ReAct) comes from the 2022 paper that formalized this pattern for LLM-based agents.

Most agent failures happen at step 1: goal specification. If the goal is ambiguous, the agent drifts. Specify success criteria, not just the task. Instead of "research competitors," say "produce a table of 5 competitors with name, pricing, and differentiator — verified against their public website as of today."

Chapter 02

The Autonomy Spectrum

1:30 – 4:00

Marina places agents on a spectrum from fully human-controlled to fully autonomous:

FULLY CONTROLLED ←─────────────────────────────→ FULLY AUTONOMOUS

  Single LLM    Fixed Pipeline    Human-in-Loop    Full Autonomy
   call only     (no branching)   (approval gates)  (acts alone)

Step-by-Step: Choosing Your Autonomy Level

  1. Define the cost of failure. Can an error be undone cheaply? If yes → lean autonomous. If no → add human gates.
  2. Identify irreversible actions. Any action that cannot be undone (send email, delete file, charge a card) requires human approval at minimum.
  3. Estimate task complexity. Simple tasks with well-defined outputs → automate fully. Complex tasks with many decision branches → add reflection loops and human checkpoints.
  4. Start conservative. Begin with human-in-loop. Verify the agent's decisions are correct over time. Remove gates one at a time as confidence grows.
  5. Never remove all gates at once. Production autonomy is earned step by step, not granted upfront.

The Devin Pattern (Marina's Reference)

The agent creates a Game Plan (numbered steps with success criteria) → human reviews the plan → agent executes. The human reviews the plan, not the outputs. Catching misalignment at the plan level costs nothing; catching it at the output level costs the full execution.

Chapter 03

Context Engineering — The Foundation of Agent Intelligence

6:45 – 9:00

"Context engineering is when you decide what information the agent has. This includes things like the background of the task, the agent's role, memory of past actions, and available tools... It's not the model alone, it's how you engineer the context around it. That's the practical foundation of intelligence in agents."

Context engineering is the highest-leverage skill in agent building. The model is fixed — you cannot change it. The context is yours to control. Everything the agent knows, believes, and can do comes from the context you construct.

What Goes Into Context

  1. System prompt / role: Who is the agent? What are its responsibilities? What are its constraints?
  2. Task / goal: What is being asked right now, with success criteria?
  3. Memory of past actions: What has the agent already done? Load last N actions from session log.
  4. Available tools: Names, descriptions, input schemas — only tools the agent can actually call.
  5. Relevant knowledge: Background information needed to complete the task accurately.
  6. Examples: Few-shot examples of good outputs or decisions — often the single highest-leverage addition.
  7. Constraints: What the agent must never do: safety rails, format requirements, cost limits.

The Context Budget Problem

With a large system prompt + task + memory + tools + knowledge, you can burn 30,000 tokens before any work starts. Solutions:

  1. Selective injection: Load only the relevant memory slices — last N entries matching the current task type.
  2. Summarization: Compress old memory into summaries. Keep recent memory verbatim.
  3. Tool registry references: Reference a tool name; let the agent look up its schema on demand, rather than inlining all schemas into every prompt.
  4. Hierarchical context: Load light context first. If the agent needs more, it requests it via a tool call.
Chapter 04

Task Decomposition — The Most Important Skill

7:13 – 10:00

"Figuring out these tasks is arguably the most important thing you'll learn about building agents. Start with how you'd do the task. Then, for each step, ask, 'Can an LLM do this?' If the answer is no, split it smaller until it is... Each step is small, checkable, and clear. When the output isn't good enough, you know exactly what step to improve."

The core rule: If you cannot identify which step failed when the agent fails, your decomposition is wrong.

Marina's Essay Decomposition Example

  1. Generate an outline (LLM)
  2. Generate search terms for each section (LLM)
  3. Run web searches (tool call — deterministic)
  4. Fetch and parse the relevant pages (tool call)
  5. Draft each section based on search results (LLM)
  6. Self-critique the draft: accurate? cited? appropriate length? (LLM)
  7. Revise based on the critique (LLM)
  8. Format and finalize (LLM or template)

Step-by-Step: Decomposing Any Task

  1. Write out how YOU would do the task. Step by step, as a smart human would. Don't think about LLMs yet.
  2. For each step, ask: "Is this one clear thing? Can I tell in 30 seconds whether it succeeded?"
  3. For each step, ask: "Can an LLM reliably do this, or does it require deterministic logic?" — LLM: generate, summarize, classify, draft, evaluate. Tool: search, fetch, calculate, write, call API.
  4. Split any step that is more than one thing. "Research and summarize" = two steps.
  5. Split any step where failure is opaque. If you can't tell which sub-task failed, split further.
  6. Define success criteria for each step. Not "did it run" but "did it produce output X in format Y."
  7. Map dependencies. Which steps must complete before others can start? Sequential dependencies = separate queue jobs.
  8. Set a timeout for each step. A step that runs forever is not a step — it's a runaway process.

Collapsing multiple steps into one LLM call produces output that's impossible to debug. When the monolith fails, you don't know if research failed, if implementation failed, or if verification failed — only that everything is wrong.

Chapter 05

Tool Use — Interface vs. Implementation

15:51 – 19:00

"Every tool has two parts: the interface for the agent — a tool name, a plain English description, and a typed input schema — and the implementation code. The agent only sees the interface. Maintain an internal registry of vetted tools with docs, versions, and ownership."

Interface (agent sees)Implementation (hidden)
NameSQL queries, API calls
When-to-use descriptionAuthentication, credentials
Typed input schemaRetry logic, rate limiting
Output format + error codesError normalization, response parsing

Step-by-Step: Building a Good Tool

  1. Name it for when to call it, not what it does. search_recent_news not query_api.
  2. Write the description as a decision rule. "Use this when you need events from the last 7 days. Do not use for historical research."
  3. Define the input schema strictly. Every field typed. Required fields marked. Include constraints ("query must be under 200 characters").
  4. Define the output schema. Success AND error look the same structurally — the agent can always parse the response.
  5. Handle every error internally. Never return a raw stack trace. Normalize: {success: false, error: "RATE_LIMITED", retry_after: 5}.
  6. Add async support. Long-running tools return a job ID immediately. Agent polls for completion. Keeps the agent loop responsive.
  7. Build a tool registry. Every tool documented with name, description, schema, error codes, version, owner. Agents load registry before deciding what to call.
  8. Test tools in isolation. Before wiring to an agent: valid inputs, invalid inputs, edge cases. Verify outputs match documented schema.
Chapter 06

Memory and Knowledge — Architecture and Separation

10:39 – 15:00

"Memory is dynamic and is updated on each run. Knowledge is static — reference material you load up front. You give it to the agent once and it can pull from that library whenever it needs to cite something accurately."

TypeWhat It IsStorageUpdates When
In-contextEverything in current prompt windowModel's active contextPer prompt
EpisodicRecords of past actions and eventsFiles, databasesAfter each action
SemanticDistilled facts and knowledgeKnowledge base, embeddingsAfter consolidation
ProceduralHow to do things — skills, protocolsSystem prompt or loaded filesWhen protocols are updated

Step-by-Step: Designing Your Memory Architecture

  1. Separate memory from knowledge at the file/database level. Memory is writable at runtime. Knowledge is read-only at runtime.
  2. Define what goes in each layer: In-context = current task + last 3 actions + relevant tool schemas. Episodic = every action (timestamped, structured). Semantic = durable facts, learned patterns. Procedural = protocols, playbooks, system prompts.
  3. Build a promotion pipeline. Episodic events → nightly consolidation extracts durable facts → semantic layer updated → next session starts with correct knowledge.
  4. Make semantic updates atomic. Don't write to knowledge base mid-cycle. Queue updates; apply after session completes.
  5. Add staleness checks. Semantic facts older than X days without re-verification get flagged. Don't let stale facts silently persist.
  6. Use vector stores for retrieval at scale. Store knowledge as embeddings. Retrieve only the relevant slice at query time — not the entire document.
  7. Version your knowledge. When the knowledge base changes significantly, bump a version. Agents that read the old version know to re-load.

RAG (Retrieval-Augmented Generation)

  1. Chunk the source documents (overlap chunks at boundaries)
  2. Embed each chunk using a consistent embedding model
  3. Store embeddings + original text in a vector database
  4. At query time: embed the query, retrieve top-N chunks by cosine similarity
  5. Inject chunks into the prompt as context
  6. Agent cites the chunks in its response
Chapter 07

Reflection and Self-Critique

22:00 – 24:00

Reflection adds a quality gate after initial generation. The agent produces an answer, then evaluates its own answer before returning it — an internal code review.

Step-by-Step: Adding Reflection

  1. Define what "good" looks like. Create an explicit rubric: accuracy, completeness, format compliance, citation quality, length.
  2. Run the primary generation. Agent produces initial output.
  3. Run the reflection pass. Feed primary output + rubric to an LLM (can be a cheaper model). Ask it to identify specific issues.
  4. Parse the reflection output. Extract the list of issues.
  5. Branch on severity. Minor issues → revise inline. Major issues → restart from step 2 with issues injected as context.
  6. Set a max revision count. Never let the reflection loop run forever. 2–3 revision cycles maximum.
  7. Log both outputs. Keep the original AND revised output. This lets you measure whether reflection actually improves quality.
  8. Measure the delta. Reflection adds latency and cost. Test with and without. If scores don't improve meaningfully, remove it.

"Make sure to test with and without reflection to ensure it's actually helping."

Chapter 08

Guardrails — Keeping Agents Safe

19:30 – 22:00

Step-by-Step: Implementing Guardrails

  1. List every action your agent can take. For each action, classify: reversible (edit file) vs. irreversible (send email, delete, charge).
  2. Gate all irreversible actions. Every irreversible action requires a human confirmation step OR a second LLM to verify before execution.
  3. Add input validation to every tool. validate_input() runs before the tool's main logic. Fail fast.
  4. Add output validation to every generation step. Every LLM call output passes through a schema check before being used as input to the next step.
  5. Implement a circuit breaker. If an agent makes the same tool call 3 times in a row with the same inputs and same output → halt and alert. This is the dead-loop pattern.
  6. Log every guardrail trigger. What triggered it, what the input was, what action was prevented.
  7. Review guardrail logs weekly. Frequent triggers on the same pattern = either the guardrail is wrong or the agent has a systemic flaw.
Chapter 09

Evaluation — Measuring Agent Quality

24:30 – 27:00

"It's important that you start evaluating right away, but also that you don't worry about having a perfect evaluation system from the get-go. You can get something working quickly and iterate over time."

Evaluation Hierarchy (Cheapest to Most Expensive)

MethodCostSpeedUse When
Code-based checksFreeInstantSchema, fields, length validation
LLM-as-judgeLowFastQuality scoring against rubric
Human evaluationHighSlowGround-truth calibration
A/B testingHighVery slowProduction traffic comparison

Step-by-Step: Building Your Evaluation System

  1. Start with the simplest meaningful check. For a summarization agent: does the output cite the source? Is it under the length limit?
  2. Define your ground truth. What does a perfect output look like? Create 5–10 labeled examples.
  3. Build an LLM judge. Write an evaluation prompt that scores outputs 1–5 against your rubric. Run on a sample set.
  4. Calibrate the judge. If LLM judge disagrees with human labels on >20% of cases, revise the evaluation rubric.
  5. Create an eval dataset. Minimum 20 examples covering the full range of task types.
  6. Run evals before and after every significant change. Never deploy without running your eval set first.
  7. Track scores over time. Plot evaluation scores by date. Downward trends = regression.
  8. Expand the eval set as you find failure modes. Every production failure = add a new eval case.

The GVU Pattern (Generator-Verifier-Updater)

Define ONE ground-truth metric per agent that cannot be gamed. This is the primary evaluation signal. Examples: coding agent → test pass rate. Research agent → citation validity rate. Trading agent → signal accuracy vs. actual market outcomes. Without a verifiable signal, self-improvement is circular.

Chapter 10

Planning — Multi-Step Execution

27:00 – 29:00
Plan-then-ExecuteInterleaved Planning
Task is well-understoodTask outcome depends on intermediate results
Steps are independentEach step informs the next
Plan can be human-verified upfrontHigh variance in execution paths
Lower latencyHigher quality (adapts to new information)

Step-by-Step: Building a Planning System

  1. Define the goal with explicit success criteria. The plan should be derivable from the success criteria.
  2. Generate the plan. Prompt the LLM to produce a numbered list of steps, each with: what happens, what tool is called, what success looks like.
  3. Validate the plan. Check each step is in the agent's capability set. Remove any step requiring tools the agent doesn't have.
  4. Present for human review (semi-autonomous systems only). This is the Devin gate.
  5. Execute step by step. For each step: run the action, collect the observation, check against the step's success criteria.
  6. Handle step failures gracefully. Retry 1–2 times. If still failing → generate a recovery plan using the error as context.
  7. Update the plan as you learn. If an early step reveals new information that changes later steps, revise the remaining plan before proceeding.
  8. Record the full trace. Every planned step, actual execution, result, and deviations. This is your post-mortem data.
Chapter 11

Multi-Agent Systems — Orchestrators and Subagents

19:35 – 23:44

"Define interfaces, not vibes. Each agent needs a clear schema for inputs and outputs. Handoffs break more often than your models do. If your researcher returns an unstructured blob and your designer doesn't know how to parse it, the whole system is going to fail."

Multi-agent adds: coordination overhead, inter-agent communication latency, resource conflicts, and new failure modes. Use it when tasks have genuine parallelism, specialization creates quality improvement, or scale requires it.

Step-by-Step: Designing a Multi-Agent System

  1. Map the task to subdomains. What are the natural boundaries? Research vs. implementation vs. verification.
  2. Define each agent's scope. What is it responsible for? What is it NOT responsible for?
  3. Define the interface between agents. What does the orchestrator send to each subagent? What does each subagent return? Every handoff is a typed contract — never unstructured text.
  4. Prevent resource conflicts. If two agents might touch the same file: one gets write access, one gets read-only. Define this in permissions, not in the prompt.
  5. Implement a coordination layer. Orchestrator tracks: which subagent is working on what, expected completion time, what to do if a subagent times out.
  6. Test each subagent independently before wiring them together. Confirm each produces the correct output schema for a range of inputs.
  7. Test the full orchestration with synthetic failures. What happens if subagent 2 fails after subagent 1 succeeds?

"If you have a simple task, skip multi-agent systems. They can slow things down and make debugging more difficult."

Chapter 12

Advanced Task Decomposition Patterns

26:14 – 28:00
Pattern 1

Functional

Split by domain expertise. Each agent handles a different domain of knowledge.

Pattern 2

Spatial

Split by area of the system. Each agent owns a different directory, service, or data store.

Pattern 3

Temporal

Split by sequence. Each phase depends on the prior. Phase B cannot start without Phase A's output.

Pattern 4

Data-Driven

Split by data partition. Each agent processes a different slice simultaneously.

Choosing the Right Pattern

  1. Is the task sequential? (each step depends on the prior) → Temporal decomposition
  2. Does the task require different types of expertise? → Functional decomposition
  3. Does the task span too much data for one context? → Spatial or data-driven decomposition
  4. Can subsets run in parallel? → Data-driven decomposition
  5. Is there a mix? → Nest the patterns (temporal outer, functional inner)
Chapter 13

Latency, Cost, and Production Tradeoffs

29:00 – 33:00

You cannot optimize quality, latency, and cost simultaneously. The right tradeoff depends on the use case.

Cost Reduction Strategies

  1. Model tiering. Cheaper models for simple subtasks (classification, extraction). Expensive models for generation and complex reasoning.
  2. Prompt caching. Cache system prompt and static knowledge. Pay once, then 10% per hit instead of 100%.
  3. Output caching. For deterministic queries, cache results. Don't re-run expensive calls for processed inputs.
  4. Batch processing. Group multiple small tasks into a single prompt instead of N separate API calls.
  5. Early stopping. Gate: "Is there actually anything new to process?" If not, exit before making any LLM calls.

Chapter 14

Observability — Log WHY, Not Just WHAT

34:00 – 35:00

"You'll want to log not only what an agent did, but why it did it. For example: 'agent chose web search instead of RAG because query contained recent' or 'reflection pass identified three issues: missing citation, vague date, wrong tone.'"

Zoom-In vs. Zoom-Out Observability

Zoom-in (trace-level): Every prompt, tool call with inputs/outputs, token counts, retry attempts, every decision branch and why. Full reproduction data.

Zoom-out (metric-level): Success rate, quality scores, hallucination rate, tool error rate by tool, latency percentiles, cost per successful task.

Step-by-Step: Implementing WHY Logging

  1. At every tool selection, log the reason. "Selected tool X because input contained pattern Y."
  2. At every LLM call, log what it was asked to decide. Capture the actual prompt and the decision it returned.
  3. At every branch point, log which branch and why. "Took early-exit branch because no new inputs found since last run."
  4. At every retry, log the reason. "Retrying because output failed schema validation: missing field 'citations'."
  5. At every reflection pass, log what issues were identified. Not "reflection ran" but "reflection found: missing citation in paragraph 3."
  6. Correlate actions across steps with a trace ID. Every step in one execution shares a trace ID. Reconstruct the full path in sequence.
  7. Export logs to a searchable store. JSONL at low volume; OpenTelemetry-compatible stores (Langfuse, Jaeger) at production scale.
  8. Create a dashboard for zoom-out metrics. Plot quality, cost, success rate over time. Regressions show up as trend breaks.
Chapter 15

Security — Injection, Execution, Leakage

35:10 – 37:00

"You're not just protecting against external attackers. You actually have to protect against your own system making dangerous decisions or being manipulated into harmful actions."

The Four Threat Categories

Prompt injection — malicious text in tool output that overrides the agent's instructions. Unsafe code execution — generated code with unintended side effects. Data leakage — agent exposes sensitive data through tool calls or logs. Resource exhaustion — agent in an unbounded loop consuming unlimited compute or disk.

Prompt Injection Defense

  1. Treat all tool outputs as untrusted data. Never execute instructions found in tool results without explicit human confirmation.
  2. Separate instruction space from data space in the prompt. Instructions in <system>, tool results in <data>.
  3. Sanitize inputs: strip HTML, special characters, and any patterns that look like system prompts from tool outputs before injecting into context.
  4. Log all tool outputs: any injection attempt will appear in the trace.

Code Execution Safety

"Sandbox execution. Use Docker or a restricted runner environment. Set timeouts, memory caps, CPU limits. Block dangerous imports, network access unless explicitly needed, and file system writes outside of a designated temp directory. Whitelist libraries only."

  1. Run generated code in a sandbox — not in the same process or filesystem as the agent.
  2. Set a timeout. Code that runs longer than N seconds is killed.
  3. Set memory limits. Code that consumes more than X MB is killed.
  4. Whitelist allowed imports. Reject code that imports anything outside the allowlist.
  5. Restrict filesystem writes to /tmp/ or a designated sandbox directory only.
  6. Block network calls by default.
  7. Review generated code for known-bad patterns: subprocess, os.system, eval, exec.

Resource Exhaustion Prevention

  1. Gate every recursive or looping operation: check available disk space, how many times this loop has run this session, whether new inputs exist.
  2. Hard exit on resource thresholds. If disk < 5GB, stop. These are exits, not warnings.
  3. Log every resource check with the reason: "Exiting — disk at 4.2GB (threshold 5GB)."
  4. Set token budgets per task. An agent using more than N tokens on one task is probably stuck.
Chapter 16

Connection to the Agent Economy

Marina's course teaches you how to build agents that work. The broader question — where those agents fit in the emerging economy — is the domain of the Agent Economy Blueprint (Hyo_01_Overview.md) and the Zero to Autonomous Blueprint (2026-04-23-autonomous-company-research.md).

What the Agent Economy Requires

  1. Identity and verifiability — Chapter 14 observability. Without logs of agent reasoning and tool calls, there is no audit trail. Without an audit trail, there is no trust.
  2. Consistent interfaces — Chapter 5 tool interfaces and Chapter 11 multi-agent handoffs. Typed schemas are the prerequisite for marketplace participation.
  3. Predictable behavior — Chapter 6 memory architecture and Chapter 9 evaluation baselines. Without these, agents cannot be trusted with consequential tasks.
  4. Resource accountability — Chapter 13 and 14 observability. Agents must report their resource consumption to bill accurately and be audited.
  5. Safety at scale — Chapter 15. Security properties become prerequisites for participation, not best practices.

Key Findings from the Zero to Autonomous Blueprint (65+ Sources)

  1. Outcome monitoring, not activity monitoring. Agents that produce perfect-looking logs while doing nothing are impossible to catch without checking expected output exists at the expected location.
  2. Per-agent ground-truth verifiers (GVU). Every agent needs one metric that cannot be gamed. Without it, self-improvement is circular.
  3. Topology discipline. UC Berkeley MAST paper: 17x error amplification when agents lack structured topology. Chapter 11 typed interfaces are the direct mitigation.
  4. HITL at plan level, not output level. Devin pattern: catch misalignment at game plan review, not after full execution. Cheapest point to intervene.
  5. Prompt caching is the single highest-ROI optimization. 90% cost reduction by caching system prompts and static knowledge across calls.

Quick Reference — Decision Framework for Any Agent

1
GOAL: Is it specific? Does it have success criteria? If NO → clarify before building anything.
2
AUTONOMY: What actions are irreversible? All irreversible → human gate required.
3
CONTEXT: Role + Task + Memory (last 3) + Tools + Knowledge. Is all of this in the prompt?
4
DECOMPOSITION: Can you tell which step failed when it fails? If NO → decompose further.
5
TOOLS: Does each tool have name, description, schema, error handling? If NO → incomplete, do not wire to agent.
6
MEMORY: Is static knowledge separate from dynamic memory? If NO → separate before running multi-agent.
7
EVALUATION: What is the ground-truth metric? If you can't define it → agent cannot self-improve.
8
REFLECTION: Did you test quality with and without it? If NO → don't add until you can measure the benefit.
9
OBSERVABILITY: Does every decision log its reason (WHY)? If NO → post-mortems are impossible.
10
SECURITY: Is generated code sandboxed? Are resource gates in place? If NO → DO NOT run autonomously.

Marina's Most-Cited Principles

Context engineering is the practical foundation of intelligence in agents.

Each step is small, checkable, and clear. When the output isn't good enough, you know exactly what step to improve.

Define interfaces, not vibes.

Log not only what an agent did, but why it did it.

Sandbox execution. Set timeouts, memory caps, CPU limits.

Start evaluating right away. Don't wait for a perfect evaluation system.

If you have a simple task, skip multi-agent systems.

Test with and without reflection to ensure it's actually helping.

Marina prepared 150 pages of research notes before recording this course, distilled from production experience building agents at Amazon. This document reads the course as a diagnostic against the Hyo system — where does our implementation match what Marina teaches, where does it diverge, and what does that divergence cost us? Seven concrete implementation changes were identified and shipped.

Finding 1: Context Engineering Is the Foundation

FINDING 01

We don't inject agent context before improvement calls

"Context engineering is when you decide what information the agent has. This includes things like the background of the task, the agent's role, memory of past actions, and available tools... It's not the model alone, it's how you engineer the context around it." — Marina Wyss, 6:45

Applied to Hyo: agent-execute-improvement.sh sends a Claude API call with a ticket description and nothing else. No PLAYBOOK.md. No ACTIVE.md. No evolution history. No GROWTH.md weaknesses. No record of what was tried last cycle. Every improvement cycle starts cold — the agent doesn't know its own failure modes, what was tried before, or what files are relevant.

Consequence: Improvement tickets retry the same approaches after failure because the agent isn't being told what failed. This is the exact context engineering failure Marina warns against.

Change 1: Before every improvement API call, inject the agent's PLAYBOOK.md + last 3 evolution.jsonl entries + ACTIVE.md current state.

Finding 2: Task Decomposition at Wrong Granularity

FINDING 02

Six steps collapsed into one 60-second inference call

"Figuring out these tasks is arguably the most important thing you'll learn about building agents. Each step is small, checkable, and clear. When the output isn't good enough, you know exactly what step to improve." — Marina Wyss, 7:13

Applied to Hyo: Our runners collapse research → implementation → verification into one monolithic call. When it fails, the exit code is the only diagnostic. We don't know which sub-step failed. This is why the same improvement tickets fail repeatedly — we don't know what specifically broke.

Change 2: Split improvement into 3 separate queue jobs. Phase A: research. Phase B: implement (reads Phase A output). Phase C: verify. Each has its own timeout. Phase B only starts if Phase A produced output.

Finding 3: Tool Interface Separated from Implementation

FINDING 03

No typed tool registry — agents guess at tool behavior

"Every tool has two parts: the interface for the agent — a tool name, a plain English description, and a typed input schema — and the implementation code. The agent only sees the interface. Maintain an internal registry of vetted tools with docs, versions, and ownership." — Marina Wyss, 15:51

Applied to Hyo: Our agent tools are shell commands. The "interface" is unstructured --help output. When an agent calls kai push sam "message", it receives either success or an unstructured error string — both may return exit code 0. Agents cannot distinguish a successful push from "HYO_HQ_PASSWORD not set — skipping."

Change 3: agents/tools.json registry — every callable tool documented with name, description, typed input schema, expected output format, error codes. Now wired into all 5 runners. SHIPPED

Finding 4: Memory vs. Knowledge Conflation

FINDING 04

KNOWLEDGE.md is being used as both static knowledge and dynamic memory

"Memory is dynamic and is updated on each run. Knowledge is static — reference material you load up front. You give it to the agent once and it can pull from that library whenever it needs to cite something accurately." — Marina Wyss, 10:39

Applied to Hyo: KNOWLEDGE.md is written to nightly by consolidate.sh. An agent running at 02:00 MT reads a different KNOWLEDGE.md than an agent running at 01:10 MT. The "static knowledge layer" has different content depending on when the agent runs — undermining the consistency Marina says it should provide.

Change 4: KNOWLEDGE.md treated as read-only during execution. Nightly consolidation still runs, but agents flag reads with a staleness check — log warning if last consolidation was >36h ago.

Finding 5: "Define Interfaces, Not Vibes"

FINDING 05

Agent handoffs are freeform markdown — not machine-parseable

"Define interfaces, not vibes. Each agent needs a clear schema for inputs and outputs. Handoffs break more often than your models do. If your researcher returns an unstructured blob and your designer doesn't know how to parse it, the whole system is going to fail." — Marina Wyss, 23:44

Applied to Hyo: dispatch report was supposed to close the loop between agents and Kai. What actually happens: agents append to markdown files using heredoc syntax, and Kai reads whatever accumulated. No schema. No required fields. No type validation. The 60 claude-delegate-failed-*.txt artifacts were the concrete evidence — each a failed delegation with no defined recovery schema.

Change 5: Structured dispatch report schema — required JSON fields: agent, cycle_id, phases_completed, outputs_written, errors, next_cycle_intent. All 5 runners now export DISPATCH_SR_* typed fields before reporting. SHIPPED

Finding 6: Log WHY, Not Just WHAT

FINDING 06

Logs record outputs — not reasoning

"You'll want to log not only what an agent did, but why it did it. For example: 'agent chose web search instead of RAG because query contained recent' or 'reflection pass identified three issues: missing citation, vague date, wrong tone.'" — Marina Wyss, 34:00

Applied to Hyo: When Ra dead-loops, the log shows repeated identical outputs but not the reasoning chain that caused the repetition. Post-mortems are impossible beyond "something ran repeatedly." Marina's zoom-in observability also requires: full trace of prompts, tool calls, token usage, retry attempts, every decision point.

Change 6: WHY logging in every agent runner — after every tool selection and every LLM call, log the reason. Prompt + token logging (model, input_tokens, output_tokens) to per-agent JSONL. SHIPPED

Finding 7: Code Execution Has No Sandbox

FINDING 07

Agent-generated code runs in production with full user permissions

"Sandbox execution. Use Docker or a restricted runner environment. Isolate code execution from your main application. Set timeouts, memory caps, CPU limits. Block dangerous imports, network access unless explicitly needed, and file system writes outside of a designated temp directory." — Marina Wyss, 35:46

Applied to Hyo: agent-execute-improvement.sh executes Claude-generated code in ~/Documents/Projects/Hyo with read/write access to every file. No sandbox. No memory caps. A malicious pattern in a research file that an agent reads and then acts on could cause arbitrary code execution. The 151GB disk fill was resource exhaustion — the exact threat Marina names.

Change 7: GATE 5.5 added — path traversal check, dangerous import whitelist, 60-second timeout on verify.sh. Generated code blocked from writing outside agents/<name>/. SHIPPED

Your browser just installed an AI you didn't ask for

Good morning. Today's brief leads with a consent story — specifically, what happens when a major tech company decides your device needs a 4GB AI model and doesn't really ask. From there we'll cover Bun's surprising architectural pivot, a Karpathy-inspired trend in agent configuration, and a crypto market that woke up green with one notable outlier. The Lab has two papers worth filing: one about when not to use tools in AI agents, and one about routing the right tasks to the right model size.

The Story · The 4 Gigabytes You Didn't Request

Sometime recently, Google Chrome started installing a 4-gigabyte AI model on users' devices without a visible consent prompt. The model is Gemini Nano — Google's on-device AI — and it has been quietly appearing in user data directories across machines whose owners had no idea it was there.

The Hacker News discussion lit up at 378 points and 390 comments, which is high-signal territory. The conversation split predictably: one camp argued that on-device AI inference is actually better for privacy than sending queries to the cloud, that the install is functionally harmless, that this is how modern software works. The other camp pointed out that 4GB is not a tracking pixel. It's a significant resource claim on someone else's machine, made unilaterally.

Both camps are partially right. On-device models do have real privacy advantages — your queries stay local, Google's servers don't see them. There's a genuine engineering argument for this deployment pattern. But the way it landed — silently, without a visible prompt, at 4GB — is also a real breach of something. Call it device sovereignty: the implicit understanding that software you've installed doesn't quietly reshape your storage landscape in the background.

The story isn't really about Chrome or Gemini Nano specifically. It's about a pattern accelerating across the industry. AI infrastructure is being bundled into software at the browser and OS layer without the usual install ceremonies. Apple baked AI into the OS without a separate consent flow. Windows Copilot shipped embedded. Chrome is doing the same with Nano. Each individual case has a reasonable defense. Collectively they add up to a normalization: AI just arrives now, without asking.

The hinge here isn't whether users revolt — most won't. It's regulation. The EU's AI Act and Digital Markets Act are both watching how foundational models get deployed on consumer devices, and a story like this hands regulators a concrete example to point at. If Google receives a ruling requiring explicit consent for on-device AI installs above some size threshold, every browser and OS maker has to navigate the same constraint overnight.

Also Moving · Three Things Worth Noting

Bun is rewriting its core in Rust. Bun — the JavaScript runtime that originally built its performance-critical internals in Zig, a systems language with a small but devoted following — committed to a Rust rewrite over the weekend. The commit landed on GitHub and hit Hacker News with 578 points and 409 comments, which is the kind of engagement reserved for genuinely surprising pivots. Zig was the idiosyncratic choice that made Bun interesting as a project; Rust is the pragmatic choice that might make Bun sustainable. For anyone building on Bun: the API surface stays the same, but the long-term maintenance story just got considerably more conventional. That's probably the right call. Convention is underrated in infrastructure.

Andrej Karpathy's CLAUDE.md file went viral. A GitHub repo containing a single configuration file — derived from Karpathy's observations about common LLM coding pitfalls, formatted as a CLAUDE.md that Claude Code reads at session start — hit 2,829 stars in a single day. If that number means nothing to you: most repositories take months to reach that. The fact that a single file of behavioral instructions became one of the day's top trending GitHub repositories tells you something important about where the developer conversation is. People aren't just using AI coding assistants anymore — they're systematically studying how to configure them better, trading notes, and building shared convention libraries. The scaffolding layer around AI agents is becoming its own engineering discipline.

The Lab · What AI can newly do this week

Routing agent tasks to the right model size — the AgentFloor benchmark.

A new paper (AgentFloor) addresses something anyone building multi-step AI agents eventually hits: most of what an agent does isn't actually hard reasoning. It's structured, routine, repetitive — formatting tool outputs, selecting from a short list, filling a template. These steps don't require a frontier model. But they usually get one anyway, because routing logic is hard to build and easy to skip.

AgentFloor introduces a 30-task benchmark organized as a six-tier capability ladder, from simple instruction-following at the bottom to complex multi-hop reasoning at the top. The core finding: small open-weight models handle the lower tiers reliably. Frontier models are only necessary at the upper tiers. If you're running a pipeline where every step hits a large model, you're almost certainly overpaying for the majority of calls. The right architecture routes by task type — small models for structured low-complexity steps, frontier models only where genuine reasoning is required.

When adding tools makes LLMs worse — the tool-use tax.

A second paper (Are Tools All We Need?) challenges a comfortable assumption: that giving an LLM access to external tools is always better than letting it reason natively. The finding is counterintuitive. When semantic distractors are present — queries that sound like they need a tool but don't — tool-augmented models don't consistently outperform plain chain-of-thought reasoning. The mechanism: prompt formatting overhead plus tool-selection errors can cancel out the informational gain from the tool call itself.

The practical rule this suggests is to benchmark native reasoning on any agent step before wiring it to an external tool. Tools add latency, cost, and failure modes. The model's internal knowledge is frequently underestimated. This pairs directly with AgentFloor — together they're two halves of the same insight about where agent spending actually goes.

Worth Sitting With

The Chrome story raises a question that doesn't have a clean answer: what does consent mean when AI is infrastructure?

We've already accepted that browsers update silently, that operating systems install patches without asking, that software makes decisions about your system in the background constantly. Somewhere we agreed that infrastructure gets to be invisible — the expectation of agency applies to features, not plumbing. Google is betting that AI is making that same transition, from feature to plumbing. They might be right.

But infrastructure has historically had an implicit social contract: it's invisible because it's neutral. The pipes don't have opinions about what they carry. A 4GB language model sitting on your device isn't quite neutral in that same way. It has objectives, it has a developer with interests, it's capable of doing things that passive infrastructure can't.

The consent frameworks we have — install/uninstall, opt-in/opt-out, terms of service that nobody reads — were designed for software that does what it's told. They may not be the right frameworks for software that can be instructed by parties you didn't invite into the conversation.

What does informed consent actually look like for AI bundled into your browser? That's not a rhetorical question. It's one the industry is going to have to answer, probably under regulatory pressure, probably soon.

Claude Platform Capabilities and Limitations: An Operational Assessment

What a Claude-powered CEO should know about its own capabilities. April 16, 2026.

Problem Statement

Kai (powered by Anthropic Claude) operates Hyo autonomously, but doesn't have a complete understanding of the Claude platform's actual capabilities and limitations. This creates blind spots:

  • CLI auth fails in launchd (headless) environments, but alternatives aren't fully documented
  • Context window limits cause session state loss, and optimal compression strategies aren't clear
  • Session isolation between different Cowork/CLI environments creates state inconsistency
  • Multi-model orchestration patterns (when to use Haiku vs. Sonnet vs. Opus) aren't defined
  • MCP (Model Context Protocol) capabilities aren't fully leveraged for tool integration

Result: Kai makes decisions based on incomplete information about its own platform. This session researches what Kai should know to operate optimally within Claude's constraints.

Core Claude Capabilities

1. Language Models: Haiku, Sonnet, Opus

Haiku (claude-3.5-haiku-20241022): Fastest, most cost-efficient. Ideal for high-volume tasks that don't require deep reasoning (data processing, parsing, routine searches). Context window: 200K tokens. Latency: 50-100ms. Cost: $0.80/$2.40 per MTok.

Sonnet (claude-3-5-sonnet-20241022): Balanced performance and cost. Best for general-purpose agent work, code generation, multi-step reasoning. Context window: 200K tokens. Latency: 200-400ms. Cost: $3/$15 per MTok.

Opus (claude-opus-4-20250514): Most capable. For complex reasoning, strategic decisions, novel problem-solving. Context window: 200K tokens. Latency: 1-2 seconds. Cost: $15/$45 per MTok.

Model Selection Strategy for Hyo: Kai should use Haiku for routine operational tasks (dispatch, file parsing, log analysis), Sonnet for agent coordination and code generation, and Opus sparingly for strategic planning and novel research. Multi-agent dispatch should default to Haiku unless the task requires creative reasoning.

2. Context Window and Prompt Caching

All models have 200K token context windows. Anthropic offers prompt caching: the first request with a large cached prompt pays full price, but subsequent identical requests within 5 minutes use cached tokens at 90% discount. For Hyo's 50K-line CLAUDE.md and 100KB+ ledger files, caching should reduce costs significantly.

Optimization: Every session should cache the hydration files (KAI_BRIEF, KAI_TASKS, CLAUDE.md) in the system prompt. Subsequent operations reuse the cache. Estimated monthly savings: 10-20% of token costs.

3. Tool Use (Function Calling)

Claude can call tools (functions) as part of reasoning. Available tools include file I/O, bash execution, web search, and integrations with external services. Tool use is sequential (Claude calls a tool, waits for result, then continues). This is different from true parallelism.

Limitation: Tool calls don't parallelize automatically. If Kai needs to fetch 10 files, it can't request all 10 in parallel — it must call fetch(file1), wait for response, then call fetch(file2), etc. Estimated latency: N * (tool_latency + response_latency). For 10 files at 500ms each: 5 seconds minimum.

Workaround: Batch tool calls when possible. If multiple operations can be done in a single shell command (e.g., `ls && grep && cat`), combine them rather than separate calls.

4. Claude Code CLI

Capabilities: The Claude Code CLI provides headless access to Claude with full API tool support (file I/O, bash, web search, etc.). It's ideal for automation in CI/CD and scheduled tasks.

Authentication: CLI auth uses a personal API key stored in `~/.anthropic/authtoken`. In headless environments (launchd, systemd, Docker), this works fine IF the key file exists and has correct permissions. However, if Kai is running Claude Code in a session (not CLI), the auth flow is different.

Known Limitation: CLI authentication fails in launchd if the home directory is not correctly set. The workaround is to explicitly set `HOME=/Users/kai` in the launchd plist before running CLI commands.

5. Cowork Collaborative Environment

Capabilities: Cowork is a web-based collaborative environment where Kai and Hyo work together in shared sessions. It provides real-time chat, file sharing, and tool integration (Slack, Linear, etc.).

Limitation: Cowork sessions have isolated session state. If Kai updates a file in Cowork, the changes don't automatically sync to the CLI or to file system watchers running elsewhere. Explicit syncs (via `kai sync` commands) are needed.

Session Continuity: Cowork sessions can be resumed, and context is preserved via summaries. However, summaries can lose fine-grained details. Critical: always hydrate before resuming a session. The hydration protocol (reading KAI_BRIEF, KAI_TASKS, etc.) catches drift that summaries miss.

Model Context Protocol (MCP) and Tool Integration

MCP Overview: MCP is Anthropic's protocol for extending Claude with custom tools and integrations. Tools are exposed as functions that Claude can call. MCPs run as separate processes and communicate with Claude via JSON-RPC.

Available MCPs for Hyo:

  • File I/O MCP: Read/write/delete files. Built-in, always available.
  • Bash MCP: Execute shell commands. Built-in, always available. Runs in isolated sandbox.
  • Web Search MCP: Search the web using Brave Search. Returns ranked results with links.
  • Browser Control MCP: Take screenshots, click, type, scroll in real browsers. Useful for UI automation.
  • Custom MCPs: User can create custom MCPs to expose domain-specific tools. Example: a Kalshi MCP that wraps the Kalshi trading API.

Optimization for Hyo: Consider building custom MCPs for high-frequency operations. Example: a scheduling MCP that can create/update/list scheduled tasks without the overhead of CLI invocation. This would reduce latency for dispatch operations by 50%.

Multi-Session State Management

Problem: Hyo may interact with Kai via Cowork (web), CLI (terminal), or scheduled automation (cron). Each interface may have different session contexts and state. If Kai updates KAI_BRIEF in Cowork, does the CLI session see the update?

Current Behavior: File updates ARE visible across sessions if the file is stored in shared git or shared filesystem. However, in-memory state (e.g., variables, context caches) are NOT shared. If Kai computes a decision in Cowork and then a CLI session starts, the CLI session doesn't know about the prior decision.

Solution Pattern: Always write decisions to persistent storage (git, ledger files) before session end. This ensures subsequent sessions can read the decision. The hydration protocol enforces this: every session reads the authoritative ledger files, so no decision is lost.

Session Recovery: If a session crashes or is interrupted, subsequent sessions should detect the incomplete work. Use a "session lock" mechanism: at session start, write a lock file. On clean exit, delete it. If a lock file exists at startup, the prior session crashed — investigate and recover.

Specific Limitations Kai Has Encountered

Limitation 1: CLI Auth in Headless Environments

Issue: When Kai runs via launchd (scheduled automation), authentication fails unless HOME is explicitly set. The error: "authtoken not found".

Root Cause: The CLI looks for auth in `~/.anthropic/authtoken`. In launchd, the default HOME is `/var/empty`, not the user's home. The file path resolves to `/var/empty/.anthropic/authtoken`, which doesn't exist.

Solution: In the launchd plist, explicitly set `EnvironmentVariablesHOME/Users/kai`. This ensures the HOME directory is correct and the authtoken is found.

Limitation 2: Context Window Compression

Issue: After 20+ sessions, context windows become fragmented. The hydration files (KAI_BRIEF, KAI_TASKS) grow large (50K+ lines). Each new session loads these in full, consuming tokens.

Solution: Implement aggressive compression. At the end of each session, archive old ledger entries (>30 days), compress ACTIVE.md files, and consolidate KAI_BRIEF. Use prompt caching to avoid re-reading the same files in the same session.

Limitation 3: Session Isolation and State Consistency

Issue: Cowork sessions and CLI sessions may diverge in state. If Hyo updates a task in Cowork, the CLI's next run doesn't see the update until it re-reads the file from disk.

Solution: Don't cache state in memory. Always read from disk before making decisions. Use version numbers in files (e.g., `version: 5` in KAI_TASKS.md). If a subsequent session sees a lower version, the prior session didn't complete its final write.

Recommended Hyo Operational Patterns

Pattern 1: Model Selection by Task

  • Haiku: Dispatch operations, log analysis, file parsing, routine searches. Threshold: <5 min reasoning time.
  • Sonnet: Agent coordination, code reviews, multi-file refactoring. Threshold: 5-15 min reasoning time.
  • Opus: Strategic decisions, novel research, architecture design. Threshold: >15 min reasoning time.

Pattern 2: Minimize Context Reloads

Use prompt caching for hydration files. The first request with KAI_BRIEF pays full cost; subsequent requests within 5 minutes use cached tokens. If sessions are batched (e.g., all morning reports run within 5 minutes), token cost drops 30-40%.

Pattern 3: Persistent Ledgers as Source of Truth

Store all state in persistent ledger files (JSON Lines format). Every session ends by updating the ledger. This makes state explicit, versionable, and queryable. No session's work is lost.

Pattern 4: Multi-Stage Operations

For long operations, break them into stages. Each stage reads initial state, executes work, writes results to ledger, and exits. The next stage reads from the ledger. This prevents context window overflow and allows stages to run at different times or even on different models.

Cost Optimization

Current Monthly Cost (estimated): 200-300M tokens at $3/MTok (Sonnet weighted average) = $600-900/month.

Optimization Potential:

  • Shift routine tasks from Sonnet to Haiku: -40% cost for that workload.
  • Enable prompt caching: -30% cost on hydration files (loaded every session).
  • Batch operations: -20% cost by parallelizing MCP tool calls.
  • Combined potential savings: 50% = $300-450/month.

Implementation effort: 2-3 weeks to refactor dispatch logic and implement caching.

Conclusion

Claude is a powerful platform, but Kai needs to understand its constraints to operate optimally. Key insights: use Haiku for routine work, Sonnet for coordination, Opus for strategy. Leverage prompt caching for repeated file access. Use persistent ledgers as the source of truth for state. And always hydrate before starting work — the system's reliability depends on it. With these patterns, Kai can operate autonomously and scale from the current 5 agents to 20+ without bottlenecks or state inconsistency.

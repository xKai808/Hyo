Phantom Position Reconciliation in High-Frequency Prediction Markets

Detecting and correcting phantom positions that inflate P&L metrics. April 16, 2026.

Problem Statement

AetherBot has phantom positions: open orders that remain on the books but don't correspond to actual inventory. The symptom: a position shows "yes_bid: ABSENT" on harvest, indicating the order failed to post but the bot believes it succeeded. When this propagates through multiple cycles, the bot's reported P&L includes unrealized positions that don't actually exist, inflating performance metrics.

Example: AetherBot thinks it holds 10 shares of "Will Bitcoin reach $100K?" but the order was rejected at the API level. The bot records a position entry and calculates profit based on current market price. When it attempts to harvest (close the position), it fails to find the order on the exchange and marks it ABSENT. The P&L inflates; the bot's statistics become unreliable.

Impact: Strategy evaluation becomes impossible because performance metrics include phantom profits. Risk management fails because reported position sizes don't match actual inventory. Harvest operations are unreliable.

Root Causes of Phantom Positions

Cause 1: API Response Loss

AetherBot sends a POST request to create a position. The exchange processes it (and creates the order), but the response packet is lost on the return. The bot's request times out or returns HTTP 5xx, so the bot assumes failure. But the order exists on the exchange — it's just invisible to the bot.

Cause 2: Partial Order Fills

AetherBot sends a request to buy 10 shares. The exchange fills 7 shares, then rejects the remaining 3 (e.g., due to insufficient liquidity). The bot receives a partial fill but interprets it as a total failure. It doesn't reconcile against the exchange's order book.

Cause 3: Race Conditions in Harvest

During harvest, AetherBot sends a close request for a position. Simultaneously, the exchange cancels the order (due to timeout or settlement). The bot's close request arrives to find an order that's already gone. It marks the position ABSENT instead of investigating.

Cause 4: State Machine Desynchronization

AetherBot's internal state machine (tracking OPEN, PENDING, CLOSED states) diverges from the exchange's actual state. A position marked OPEN locally is actually FILLED on the exchange, but AetherBot doesn't reconcile.

Reconciliation Patterns from Traditional HFT Systems

Pattern 1: Order Book Snapshots

After every position creation, immediately fetch the exchange's full order book snapshot. Validate that the newly created order appears in the order book with the correct size, price, and side. If it doesn't appear, log an alert and resubmit.

This catches API response loss: if the order was created but the response was lost, the order book snapshot will show it.

Pattern 2: Settlement Confirmation

Before marking a position as settled (closed), require confirmation from the exchange's settlement API. For Kalshi, this means checking the contract's final balance for the account. For example, after closing a position in "Will Bitcoin reach $100K?", verify that the account's balance in that market has been returned and the position is marked EMPTY or CLOSED.

Pattern 3: Position Reconciliation Loop

Every N minutes (e.g., 5 minutes), run a full reconciliation loop:

  1. Fetch the exchange's full position list (what does the exchange say I own?)
  2. Fetch AetherBot's internal position state (what do I think I own?)
  3. Diff: positions in exchange but not in state = PHANTOM CREATES (need to sync local state)
  4. Diff: positions in state but not in exchange = PHANTOM CLOSES (need to remove from state)
  5. For each mismatch, log to `agents/aether/logs/reconciliation-YYYY-MM-DD.jsonl` and take corrective action

Pattern 4: State Machine with Acknowledgment Gates

Instead of optimistic state updates, use acknowledgment gates:

  • INTENT: Bot decides to open position XYZ
  • REQUEST_SENT: Bot sends API request; starts a timeout timer
  • CONFIRMATION_RECEIVED: Bot receives exchange confirmation; order appears in order book snapshot
  • FILLED: Bot verifies position is in account inventory
  • SETTLED: Settlement confirmation received

A position only advances state when the next gate condition is confirmed. If a gate times out, the bot doesn't mark the position as successful.

Pattern 5: Dual-Path Confirmation

For critical positions, require dual confirmation: both a direct API call AND a check against the order book snapshot. If either fails, the position is marked UNCONFIRMED and flagged for manual review.

Implementation for AetherBot v254+

Phase 1: Order Book Snapshot Integration (3 days)

After every position creation request, fetch the order book snapshot (via Kalshi's book API) and verify the order appears within 2 seconds. If it doesn't, retry 3 times. If still missing, log as CREATION_FAILED and alert.

Store order book snapshots to `agents/aether/logs/order-book-YYYY-MM-DD.jsonl` with timestamp, order ID, and status. These become the source of truth for reconciliation.

Phase 2: Reconciliation Loop (5 days)

Add a reconciliation routine that runs every 5 minutes:

def reconcile(): exchange_positions = fetch_kalshi_account() local_positions = read_local_state() for pos in exchange_positions: if pos not in local_positions: log("PHANTOM_CREATE", pos) sync_local_state(pos) for pos in local_positions: if pos not in exchange_positions: log("PHANTOM_CLOSE", pos) remove_local_state(pos)

Run this reconciliation before every harvest. If any phantoms are detected, halt harvest and alert.

Phase 3: Settlement Confirmation (7 days)

Before marking a position as CLOSED, fetch the settlement status from Kalshi. Verify that the position balance has been returned to the account and the contract is marked as settled. Only then mark CLOSED locally.

Phase 4: State Machine Gates (10 days)

Refactor AetherBot's state machine to use the five-stage acknowledgment protocol. Each stage requires explicit confirmation. A position stays in PENDING_CONFIRMATION until all gates have passed.

Monitoring and Alerting

Log all reconciliation results to `agents/aether/logs/reconciliation-YYYY-MM-DD.jsonl` with:

  • Timestamp
  • Position ID and contract
  • Exchange state (size, price, side, status)
  • Local state (size, price, side, status)
  • Mismatch type (PHANTOM_CREATE, PHANTOM_CLOSE, SIZE_MISMATCH, PRICE_MISMATCH)
  • Corrective action taken

Alert conditions:

  • Any PHANTOM_CREATE outside harvest window → P1 alert (uncontrolled position opened)
  • Any PHANTOM_CLOSE during active trading → P2 alert (position disappeared unexpectedly)
  • Reconciliation finds >3 mismatches in 1 cycle → P1 alert (state machine desynchronized)

Expected Outcomes

  • Accurate P&L: P&L metrics no longer include phantom positions; performance reports reflect actual trades only.
  • Reliable Harvest: Harvest operations detect and correct state drift automatically.
  • Risk Management: Reported position sizes match actual inventory.
  • Early Anomaly Detection: Phantom position detection alerts to API failures or exchange desynchronization in real-time.

Validation and Testing

Test the reconciliation loop by:

  1. Manually inject phantom positions into local state
  2. Run reconciliation and verify phantoms are detected and removed
  3. Simulate API response loss (send position request but ignore response) and verify order book snapshot catches it
  4. Simulate settlement delay (close position but check settlement API fails) and verify bot waits for confirmation

Conclusion

Phantom position reconciliation is essential for reliable automated trading. Order book snapshots, settlement confirmation, and periodic reconciliation loops catch phantom positions quickly. The investment is modest (2 weeks implementation) and the payoff is substantial: accurate P&L, reliable operations, and trustworthy performance metrics.

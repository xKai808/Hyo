Dex Research Report

Automated System Remediation: Beyond Detection

Moving from detection-only integrity checking to auto-remediation. April 16, 2026.

Problem Statement

Dex detects issues (stale files, broken links, configuration drift, out-of-sync state) but cannot fix them. Current behavior: Dex runs health checks, discovers problems, logs them to `ledger/known-issues.jsonl`, and alerts via console. Humans must then manually remediate: restore a file from git, update a config, resync a database. This creates a bottleneck:

  • Issues are discovered automatically but fixed manually
  • Response time is hours to days, not minutes
  • Manual remediation is error-prone (wrong file version restored, incomplete rollback)
  • Issues can cascade if they aren't fixed quickly

For a small team running 5+ autonomous agents, the cost of manual remediation is high. A stale file that broke the build, a configuration drift that corrupted state, or a broken symlink can propagate through multiple agents before being noticed. Automated remediation closes this gap: the system detects AND fixes issues, with humans reviewing the remediation afterward.

Remediation Patterns from Infrastructure Automation

Pattern 1: Self-Healing Configuration via Ansible

Tool: Ansible (open-source, 31.5K GitHub stars, GPLv3). Ansible is an infrastructure automation engine that applies idempotent configuration changes. A playbook describes the desired state (e.g., "file X should contain Y", "service Z should be running"). When run, Ansible checks the current state, and if it differs from desired, it applies corrections.

Key advantage: Ansible is idempotent. Running it 100 times produces the same result as running it once. This is crucial for automated remediation — if a fix partially fails and is retried, the system doesn't break.

For Hyo: Define Ansible playbooks for common issues. Example playbook for stale files:

- name: Restore stale files from git hosts: localhost tasks: - name: Check if file is stale stat: path=/path/to/file register: file_stat - name: Restore if stale shell: git checkout HEAD -- /path/to/file when: file_stat.stat.mtime < (ansible_date_time.epoch|int - 86400)

Pattern 2: Policy Enforcement via OPA (Open Policy Agent)

Tool: OPA (open-source, 9.1K GitHub stars, Apache 2.0). OPA is a policy engine that evaluates configuration files and enforces compliance rules. Rules are written in Rego, a high-level policy language. OPA can be integrated into CI/CD or runtime checks to automatically reject non-compliant configs.

Example: An OPA policy enforces "all agent config files must have a timeout field". When an agent config is modified, OPA checks it. If the timeout is missing, the policy fails and the config change is rejected. Humans must fix it before deployment.

For Hyo: Define OPA policies for configuration correctness:

  • All agent manifests must have a schedule field
  • All API endpoints must have rate-limit config
  • All secrets must live in agents/nel/security/ only

Pattern 3: Automated Rollback via Argo Rollouts

Tool: Argo Rollouts (open-source, 4.8K GitHub stars, Apache 2.0). Argo Rollouts is a progressive delivery tool that automates canary deployments and automatic rollback. When a new version is deployed, Argo monitors metrics. If metrics degrade (e.g., error rate spikes), it automatically rolls back to the previous version.

For Hyo: Deploy agent updates progressively. Monitor key metrics (error rate, run duration, memory usage). If any metric degrades >10%, automatically roll back to the prior version and alert.

Pattern 4: Self-Correction via Declarative State Management

Instead of imperative fixes ("run this script to fix X"), declare the desired state and let a controller continuously enforce it. Example:

# agents/dex/remediation/desired-state.json { "symlinks": { "website/": "agents/sam/website/", "newsletter/": "agents/ra/pipeline/" }, "file-permissions": { "agents/nel/security/": "mode 700" }, "file-freshness": { "KAI_BRIEF.md": "max_age_seconds 3600", "agents/*/ACTIVE.md": "max_age_seconds 86400" } }

A controller runs every 10 minutes and enforces this state: checks symlinks are correct, permissions are right, files are fresh. If any divergence is detected, the controller automatically corrects it and logs the action.

Implementation for Hyo's Dex Agent

Layer 1: Detection with Prescribed Remediation (Week 1)

Enhance Dex's health checks to include a "remediation script" field. When an issue is detected, Dex doesn't just log it — it runs the associated remediation script (with safeguards). Example:

issue = { "type": "stale_file", "file": "KAI_BRIEF.md", "remediation": "git checkout HEAD -- {file}", "safeguard": "backup {file} before restore" }

When detected, Dex: (1) backs up the file, (2) runs the remediation script, (3) logs the action, (4) verifies the fix worked (e.g., file is now fresh).

Layer 2: Ansible Integration for Complex Fixes (Week 2)

For complex issues (e.g., symlink repair, permission correction), use Ansible playbooks. Dex invokes Ansible with the appropriate playbook and captures output.

Layer 3: OPA Policy Enforcement (Week 3)

Integrate OPA into the deployment pipeline. Any agent config change that violates a policy is rejected before deployment. Humans see the policy failure and must fix the config.

Layer 4: Argo Rollouts for Agent Updates (Week 4)

Deploy agent code changes progressively. Monitor agent health (CPU, memory, error rate, run duration). If any metric degrades, automatically roll back to the prior version.

Remediation Playbooks for Hyo

Playbook 1: Stale Files

  • Detect: File mtime > 24 hours old
  • Remediate: `git checkout HEAD -- {file}`
  • Verify: File mtime is now recent

Playbook 2: Symlink Drift

  • Detect: Symlink target doesn't match expected target
  • Remediate: `ln -sf {expected_target} {link}`
  • Verify: `readlink {link}` matches expected

Playbook 3: Permission Drift

  • Detect: File permissions don't match expected (e.g., agents/nel/security/ is world-readable)
  • Remediate: `chmod {expected_mode} {file}`
  • Verify: `stat {file}` shows correct mode

Playbook 4: Agent Stale State

  • Detect: Agent's ACTIVE.md is older than 48 hours
  • Remediate: Trigger an agent health check to update ACTIVE.md
  • Verify: ACTIVE.md mtime is fresh

Playbook 5: Config Invalid

  • Detect: Agent config fails JSON schema validation
  • Remediate: Restore config from git HEAD; alert human if no valid prior version exists
  • Verify: Config passes schema validation

Safeguards and Verification

Automated remediation is powerful but dangerous. Safeguards are essential:

  • Dry-Run First: Before applying a remediation, run it in dry-run mode to show what would change
  • Backup Before Modify: Before overwriting a file, back it up to `agents/dex/remediation-backups/{timestamp}/`
  • Verification Step: After remediation, verify the fix actually worked (e.g., file is now fresh, symlink is correct)
  • Audit Log: Log every remediation action to `agents/dex/logs/remediation-YYYY-MM-DD.jsonl` with: timestamp, issue type, remediation script, output, success/failure
  • Rollback Plan: Every remediation must have a rollback script (e.g., "restore config from backup")
  • Human Review: High-risk remediations (e.g., permission changes, file deletions) require human approval before execution. Medium-risk remediations (e.g., file restore from git) execute automatically but are logged for review. Low-risk remediations (e.g., symlink repair) execute without approval.

Expected Outcomes

  • Faster Recovery: Issues are fixed in minutes, not hours or days.
  • Fewer Manual Interventions: Humans review fixes after they're applied, not before.
  • Fewer Cascading Issues: Rapid remediation prevents problems from propagating through agents.
  • Reliable Self-Healing: The system keeps itself in a valid state automatically.
  • Auditable Changes: Every remediation is logged, backed up, and reversible.

Risk Mitigation

Automated remediation has risks. Mitigation strategies:

  • Start with low-risk remediations only (symlink repair, file freshness). Add high-risk remediations (permission changes, deletions) only after 2 weeks of zero false positives.
  • Always maintain a rollback capability. Every backup is tested (periodically restore from backup and verify it works).
  • Alert on every remediation, even if successful. Humans see the audit log and can investigate anomalies.
  • Rate-limit remediation attempts. Don't run more than 5 remediations per minute. If the system is broken badly enough to need constant fixing, halt and alert rather than entering a repair loop.

Conclusion

Automated remediation transforms Dex from a reporting tool into a self-healing system. Ansible, OPA, and Argo Rollouts provide the tooling. Implementation over 4 weeks brings Hyo's system uptime and reliability in line with production infrastructure. The result: a small team can reliably operate a complex multi-agent system without manual intervention.

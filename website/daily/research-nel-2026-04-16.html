Nel Research Report

Adaptive Security Monitoring for AI Agent Systems

Moving from static scanning to behavioral anomaly detection in agent-driven architectures. April 16, 2026.

Problem Statement

Nel's current security posture relies on static analysis — gitleaks for credential scanning, trufflehog for token detection, and manual code review for permission boundaries. While effective for catching hard-coded secrets, this approach has fundamental blind spots:

  • No detection of logic bugs where an agent accesses a resource it shouldn't based on state
  • No visibility into runtime behavior — what an agent actually does, only what code appears to do
  • No continuous monitoring — security checks happen at commit time, not during execution
  • No anomaly detection — unusual API call patterns, privilege escalation, or lateral movement go unnoticed
  • Manual CVE tracking — no continuous vulnerability scanning of dependencies

For Hyo's architecture, where Nel, Ra, Sam, Aether, and Dex execute autonomously on a schedule, static scanning is insufficient. An agent could be compromised mid-execution, or a dependency could have a newly-disclosed vulnerability that bypasses code review.

Security Monitoring Patterns in Traditional Systems

Mature security operations combine three layers:

Layer 1: Dependency Vulnerability Scanning

Tool: Trivy (CNCF, open-source, 8.2K GitHub stars, Apache 2.0). Trivy scans container images and filesystem artifacts for known CVEs in real-time. It maintains a continuously-updated vulnerability database (sourced from NVD, GitHub, and vendor advisories) and can be run during CI/CD or as a pre-execution gate.

For Hyo: Run Trivy on each agent's environment before launching the runner. If a critical CVE is detected in a dependency, halt execution and alert.

Layer 2: Application-Level Anomaly Detection

Tool: Falco (CNCF, runtime security, 6.1K GitHub stars, Apache 2.0). Falco monitors system calls and container behavior in real-time, detecting suspicious patterns like privilege escalation, unusual network connections, or file access outside expected baselines.

For Hyo: Define baseline behavior for each agent (Nel: filesystem reads in /kai/ledger, network calls to security scanners; Ra: HTTP requests to newsletter sources, SMTP outbound; Sam: network calls to Vercel API, file writes to website/). Anomalies trigger alerts and execution suspension.

Layer 3: API-Level Behavioral Monitoring

Tool: OWASP ZAP (15.1K GitHub stars, Apache 2.0) for penetration testing, and custom instrumentation for monitoring. ZAP can perform authenticated scanning of APIs and detect injection flaws, insecure deserialization, and broken authentication.

For Hyo: Instrument each agent's external API calls (to Kalshi, SMTP, Vercel, GitHub) with request/response logging, signature validation, and rate-limit tracking. Flag deviations from expected patterns (e.g., Ra suddenly making 1000 requests when normal is 50).

Building a Closed-Loop Security Monitoring System

The architecture should follow a feedback loop:

  1. Pre-Execution Gate: Trivy scans agent dependencies. If any critical CVE, halt and report. If pass, proceed.
  2. Baseline Establishment: On first run, Falco profiles normal agent behavior (system calls, network patterns, file access). This baseline is committed to `agents//security/baseline.json`.
  3. Runtime Monitoring: Falco watches the agent's execution. Any deviation from baseline (e.g., unexpected privilege escalation, outbound connection to unfamiliar IP) triggers an alert.
  4. API-Level Logging: All outbound API calls are logged with method, endpoint, request size, response size, latency. Anomalies (e.g., 10x larger request than normal, unusual endpoint access) trigger alerts.
  5. Post-Execution Analysis: After the agent completes, Nel performs OWASP ZAP scanning on any APIs the agent exposed or modified. Results are compared to prior scans for regression detection.

Implementation for Hyo

Phase 1: Dependency Scanning (2-3 days)

Integrate Trivy into the pre-execution gate in `kai/queue/exec.sh`. Before any agent runs, Trivy scans the agent's Python environment and any system dependencies. If a critical CVE is found, the execution is blocked and an alert is sent via `dispatch alert`.

Phase 2: Baseline Profiling (1 week)

Run each agent for one full week with Falco enabled. Capture system call patterns, network connections, and file access. Use Falco's rule-building to codify normal behavior. Store baselines in `agents//security/baseline.json`.

Phase 3: Runtime Anomaly Detection (2 weeks)

Deploy Falco with baselines. Any deviation triggers an alert. Tune false positives based on observed legitimate variance. Integrate alerts into `dispatch alert` for visibility in HQ.

Phase 4: API-Level Instrumentation (3 weeks)

Add middleware to each agent's external API calls. Log request/response pairs to `agents//logs/api-calls.jsonl`. Nel performs daily anomaly analysis on call patterns (rate, size, latency, endpoints accessed).

Expected Outcomes

  • CVE Detection: Zero-day vulnerabilities in dependencies are caught before execution, reducing blast radius to zero.
  • Runtime Anomalies: Compromise of an agent (via code injection, dependency hijacking, or privilege escalation) is detected within seconds, not days.
  • Behavioral Baselines: Normal behavior is codified; deviations are actionable and investigated.
  • API Integrity: Outbound API calls are logged and analyzed for signs of lateral movement or data exfiltration.

Conclusion

Moving from static scanning to adaptive, behavior-based monitoring is essential for autonomous agent systems. Trivy, Falco, and OWASP ZAP provide the tooling foundation. Implementation in Hyo is straightforward and can be phased over 4 weeks. The result is a security posture that matches the operational tempo: continuous, automated, and closed-loop.

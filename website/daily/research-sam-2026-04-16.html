Automated Regression Detection for Small-Scale Web Platforms

Integrating performance baselines and regression detection into Vercel deployments. April 16, 2026.

Problem Statement

Sam deploys the Hyo website to Vercel, but has zero regression detection. Current situation:

  • No performance baselines — no baseline metrics for API response time, page load speed, or Core Web Vitals
  • No automated testing post-deploy — a deployment succeeds if the build completes, not if the platform actually works
  • No regression tracking — performance degradation (e.g., a page that takes 5 seconds to load instead of 1) is invisible until a user complains
  • No early warning system — performance problems are discovered reactively, not proactively

For a small-scale platform like Hyo's website, this means regressions can accumulate undetected: dependency bloat slows page load, API endpoints degrade, or front-end changes introduce memory leaks. The cost is poor user experience and unmonitored technical debt.

The Regression Detection Stack

Layer 1: Lighthouse CI for Page Performance

Tool: Lighthouse CI (Google, open-source, 9.1K GitHub stars, Apache 2.0). Lighthouse audits web performance, accessibility, SEO, and best practices. Lighthouse CI runs Lighthouse on every deployment and compares results to a baseline. If performance degrades, the CI step fails and the build is rejected (or flagged).

Metrics Lighthouse measures: Largest Contentful Paint (LCP), Cumulative Layout Shift (CLS), First Input Delay (FID), Time to First Byte (TTFB), JavaScript execution time, CSS bloat, image optimization.

Layer 2: Web Vitals Instrumentation

Tool: web-vitals library (Google, 6.8K GitHub stars, Apache 2.0). Embed JavaScript into the website's frontend that continuously measures Core Web Vitals (LCP, FID, CLS) in production. Send telemetry to an analytics endpoint (e.g., Vercel Analytics or a custom endpoint).

This captures real-user performance, not synthetic. If a feature is slow for 50% of users but fast for 50%, Lighthouse might miss it, but real-user telemetry catches it.

Layer 3: sitespeed.io for Synthetic Monitoring

Tool: sitespeed.io (open-source, 3.9K GitHub stars, Apache 2.0). sitespeed.io runs repeated performance tests from a fixed location and compares results across deployments. It measures not just page load time, but waterfall analysis (which resources block rendering), HTTP/2 efficiency, CPU usage, memory, and network simulation (e.g., "how does this site perform on 4G?")

Run sitespeed.io after every Vercel deployment against 5-10 key pages (homepage, docs, API reference). If any page regresses >10%, block the deployment and alert.

Layer 4: API Response Time Monitoring

Tool: Vercel Analytics + custom instrumentation. Measure API response times for each endpoint. Set baseline SLOs: API endpoints should respond in <100ms for 95th percentile. Endpoint functions that exceed SLO are flagged as regressions.

Log latency to `agents/sam/logs/api-performance-YYYY-MM-DD.jsonl` with endpoint, method, status code, response time, and timestamp. Compute daily baseline (p50, p95, p99). If p95 latency exceeds baseline by >20%, alert.

Layer 5: Automated Rollback on Severe Regression

Tool: Vercel's instant rollback + GitHub Actions. Integrate the above into GitHub Actions: on every push to main, build → deploy to preview → run Lighthouse CI + sitespeed.io + API load tests. If any metric fails, rollback the deployment and block the merge.

This turns regression detection into an automated gate: a regression never reaches production.

Integration with Vercel's Deployment Pipeline

Step 1: Establish Baselines

Run Lighthouse CI, sitespeed.io, and API monitoring against the current production deployment. Capture these metrics as baselines. Store in `agents/sam/website/baselines.json`: { "lighthouse": { "performance": 85, "accessibility": 95, "best-practices": 92 }, "core-web-vitals": { "lcp": 1200, "fid": 50, "cls": 0.05 }, "api-slo": { "p95": 80, "p99": 150 } }

Step 2: GitHub Actions Workflow

Add a `.github/workflows/regression-detection.yml` that runs on every push:

  1. Build the Next.js app
  2. Deploy to Vercel preview
  3. Run Lighthouse CI against preview
  4. Run sitespeed.io against 10 key pages
  5. Run API load tests against preview endpoints
  6. Compare results to baselines
  7. If any regression >10%, comment on the PR and block merge
  8. If all checks pass, merge and deploy to production

Step 3: Real-User Monitoring

Add the web-vitals library to the frontend. Collect metrics from real users and send to a custom endpoint (e.g., `/api/telemetry`). Store in `agents/sam/logs/web-vitals-YYYY-MM-DD.jsonl`. Track p50, p95, p99 for LCP, FID, CLS daily. Flag if any metric exceeds SLO.

Step 4: Daily Regression Report

Every morning, Sam runs a regression analysis script that compares yesterday's metrics to the baseline and week-ago average. Log results to `agents/sam/logs/regression-report-YYYY-MM-DD.jsonl`. Surface regressions in the morning report on HQ.

Specific Implementation for Hyo

Lighthouse CI: Add `.lighthousecirc.json` to the website repo with thresholds:

"budgets": [ { "resourceType": "script", "budget": 800, "warnings": 750 }, { "resourceType": "style", "budget": 200 } ]

sitespeed.io: Run post-deploy via GitHub Actions. Test pages: `/`, `/docs`, `/api`, `/hyo-research`. Accept 10% regression before failing.

API Monitoring: Add timing middleware to Next.js API routes. Log to telemetry endpoint. Set SLOs: `/api/hyo-research` <100ms p95, `/api/dispatch` <150ms p95.

Expected Outcomes

  • Regression Detection: A slow deployment is caught before reaching production users.
  • Performance Baselines: Sam knows what "good" looks like and can measure against it.
  • Proactive Alerts: Performance degradation is detected in hours, not days or weeks.
  • Automated Rollback: Severe regressions trigger automatic rollback, eliminating the need for manual remediation.
  • User Experience Protection: End-users never see a broken or slow deployment.

Implementation Timeline

  • Day 1-2: Set up Lighthouse CI with baseline metrics. Integrate into GitHub Actions.
  • Day 3-5: Instrument web-vitals and API monitoring. Set SLOs.
  • Day 6-7: Deploy sitespeed.io. Test against 10 key pages.
  • Day 8: Build regression report generator. Integrate into morning report pipeline.
  • Day 9-14: Tune thresholds and false positives. Document runbook for handling regressions.

Conclusion

Regression detection transforms Sam's deployments from "it builds, ship it" to "it builds, it performs, ship it." Lighthouse CI, sitespeed.io, web-vitals, and API monitoring provide full-stack performance visibility. For Hyo's small team, a 2-week implementation yields continuous protection against performance regressions.
